## 0. 一句話結論

素材不是「都爛」，而是 **至少 4～5 套互不相容的生成世代疊在同一戰場**；再疊上 **多數高解析圖沒有可用透明通道、4 座塔與 3 種敵缺檔、神話英雄立繪兩極落差**。
以商店卡／首屏混戰截圖為準：**最大短板是風格一致性與合成就緒度（alpha／背景）**，不是單一角色「畫工不夠華麗」。

**素材健康度（主觀）**：約 **4.5 / 10**（可玩、可辨識 emoji 後備仍在；不可宣稱「完整商用美術包」）。

## 1. 審查方法與優先級定義

### 1.1 怎麼審

| 維度 | 檢查內容 |
|------|----------|
| 檔案存在 | `TOWERS` / `ENEMIES` / `HEROES` 是否有對應 PNG |
| 技術就緒 | 解析度、PixelFormat、**真透明（A=0）**、背景是否烤進圖 |
| 風格一致性 | 線稿厚度、上色、視角、細節密度是否同一宇宙 |
| 可讀性 | 縮到戰場尺寸（約 `CELL×0.5～0.8`）後剪影是否仍可辨 |
| 用途適配 | 平鋪磚 vs 裝飾物 vs 立繪 vs 板面單位是否用錯構圖 |

抽樣：全量列目錄與尺寸；視覺逐張檢視主要塔／敵／磚／英雄；以 `System.Drawing` 抽樣四角與中心 alpha。

### 1.2 優先級

| 等級 | 含義 | 建議時程 |
|------|------|----------|
| **P0** | 宣傳圖／首局一眼穿幫；缺檔；白底／實心底無法合成；神話卡池兩代落差 | 下一輪美術必做 |
| **P1** | 戰場可讀但風格割裂；基礎敵落後精英；磚平鋪／方向磚不合格 | 第二輪 |
| **P2** | 細節 polish、體積差、缺檔以外的強化、pipeline 文件化 | 有餘力再做 |

| 世代代號 | 代表檔 | 視覺特徵 | 合成 | 適合作為標竿？ |
|----------|--------|----------|------|----------------|
| **Gen-A 插畫塔** | arrow／cannon／frost／tesla | 平塗厚線、等距／3/4、**實心紫底** | ❌ | 否（底與風格皆舊） |
| **Gen-B 暗黑塔** | sniper／arcane | 高細節、金邊石材、**實心黑底** | ❌ | 塔體語言可借，須重出 alpha |
| **Gen-1 戰場單位** | slime／goblin／orc／imp／shieldman／medic／bat／frostwolf／boss；神話 chibi 四張 | chibi、線稿清晰、**真透明**、~200px | ✅ | **戰場單位基準**（要升解析度） |

#### 缺檔 4 座（P0）

| 塔 id | 名稱 | 元素色錨（`def.color`） | 為何不能只靠 emoji |
|-------|------|-------------------------|-------------------|
| `poison` | 毒霧塔 | `#22c55e` | 持續輸出主力之一，商店常駐 |
| `support` | 聖光塔 | `#fde047` | 支援塔無攻擊，**全靠圖示傳達「不開火」** |
| `beacon` | 引魂燈塔 | `#fb7185` | 揭露／減速語意需要燈籠剪影 |

#### 塔類總評

- **風格雙軌**：4 張卡通插畫 vs 2 張暗黑寫實，並排放在建塔列會直接穿幫。
- **技術**：6/6 **無真透明** → 戰場畫在草／路徑上會出現色塊底座（或需程式色鍵，現況不可靠）。
- **語意缺口**：物理系（arrow／sniper）、火系（cannon／mortar）、支援雙塔皆缺完整視覺階梯。

## 8. 統一交件規格（Style Bible + 技術）

### 8.0 鎖定美術方向（建議）

**主風格名稱**：`Eastern Dark Fantasy · Clean Game Sprite`

| 規則 | 要求 |
|------|------|
| 線稿 | 清晰深色描邊（1～2px @ 輸出解析度），禁止油畫無邊 |
| 上色 |  saturation 中高；元素色可對應 `def.color` |
| 光源 | 單一頂／前光；避免多色霓虹搶戲（升級彩虹留給程式） |
| 背景 | **禁止**紫灰插畫底、禁止白底、禁止烤黑底；只要 **透明** |
| 視角 | 塔：3/4 等距略俯；敵／英雄戰場：3/4 全身；卡面：略仰或英雄構圖全身 |
| 禁止 | 圖內文字、浮水印、UI 框、隨機多角色、照片寫實 |

**標竿參考（畫工上限）**：`zhongkui.png`、`yaksha.png`、`sniper.png` 的密度與材質。
**標竿參考（合成正確）**：`medic.png`、`slime.png`、`wukong.png` 的透明與剪影。
→ 最終產物 = **標竿畫工 + 標竿透明**。

### 8.1 塔（Tower）規格

| 欄位 | 規格 |
|------|------|
| 尺寸 | Master **1024×1024**；可另供 runtime 256 |
| 格式 | `PNG-32`，**RGBA**，四角 A=0 |
| 構圖 | 單塔置中；底座完整；主體佔高 75%±10% |
| 內容 | 無士兵站台（避免與英雄混淆）；武器／元素特徵占剪影 30% 以上 |
| 命名 | `assets/towers/{id}.png` 與 `TOWERS` id 一致 |
| 必交 10 id | arrow, cannon, frost, tesla, poison, support, beacon, sniper, arcane, mortar |

### 8.2 戰場單位（敵／基礎英雄方向圖）

| 欄位 | 規格 |
|------|------|
| 尺寸 | 敵 Master **512×512**（小兵可 256）；方向英雄 **512×512** 四向 |
| 格式 | PNG-32 RGBA |
| 構圖 | 全身；腳底留 5～8% padding；無地面陰影板（陰影交程式） |
| 可讀性 | 縮至 **48×48** 仍可辨：體型、主武器、元素色 |
| 方向圖 | down／up／left／right 服裝色、體型、武器一致；僅朝向變 |

### 8.3 英雄卡面／神話立繪

| 欄位 | 規格 |
|------|------|
| 尺寸 | **1024×1024**（或 1024×1536 直式，若 UI 固定方圖則強制方） |
| 格式 | PNG-32 RGBA |
| 構圖 | 全身或膝上；標誌物清晰；特效不蓋臉 |
| 密度 | 與 zhongkui／baisuzhen 同級；禁止再交 ~192px chibi 當傳說 |
| 雙用途（可選） | 另出 `*_battle.png` 256 簡化版供場上 |

### 8.4 地圖磚

| 類型 | 尺寸建議 | 格式 | 特殊 |
|------|----------|------|------|
| grass1–3、path fill | Master 512 或 1024；runtime 128 | PNG 可無 alpha | **seamless** 四邊可拼；path **無強制方向中線** |
| rock／bush／tree | 256 或 512 | PNG-32 RGBA | 硬邊透明；無 vignette；頂視一致 |

### 8.5 驗收清單（每張必過）

1. **存在性**：對應 id 有檔
2. **Alpha**：裝飾／角色／塔四角 A=0（地板磚除外）
3. **風格**：與 Style Bible 同批抽檢並排通過
4. **剪影**：48px 縮圖可辨
5. **語意**：元素／職業／武器不與近義單位撞車
6. **體積**：runtime 建議 < 200KB／張（master 可較大）
7. **無字**：圖內無文字／LOGO
